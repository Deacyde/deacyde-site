Question 1
Why does Short Sword (5001) work but Soul Devourer (11001) doesn't?

Both items now have identical critical flags: nodrop=1, norent=1, itemclass=0, itemtype=0, loregroup=0. Short Sword is an unmodified PEQ database item (id 5001). Soul Devourer was modified by us (originally a PEQ item, id 11001). The buy handler calls database.GetItem(item_id) which reads from shared memory — it returns the item for 5001 but null for 11001.

Key difference: Soul Devourer has proceffect=995 (a spell proc) and magic=1. Short Sword has proceffect=-1 and magic=0. Could a bad proc spell reference cause shared_memory to skip the item?

Question 2
Why do bags (containers) work with any flags?

Baggins' items (itemclass=1, itemtype=11) buy fine — even when they had nodrop=0, norent=0. Weapons/armor (itemclass=0, itemtype=0/3/4/10) fail with the same flags.

In the source code, bags use cursor=false (placed directly into inventory) while equipment uses cursor=true (placed on cursor). But the failure happens BEFORE this — at the GetItem() call. So the cursor logic shouldn't matter.

Research angle: Does the shared_memory loader or content filtering system treat itemclass=1 (containers) differently than itemclass=0 (common items)?

Question 3 — Strong Lead 🔥
Is content/expansion filtering excluding our items?

The zone boot log shows: "Loaded [939] of [1152] merchant list entries" — meaning 213 items were filtered out. EQEmu has a content filtering system that can exclude items based on expansion settings.

Server config: Expansion:CurrentExpansion = 9 (Shadows of Luclin), World:ExpansionSettings = 524287. If our items have expansion metadata that puts them beyond Luclin, they'd be filtered from merchant lists at zone load time — meaning they'd appear in the merchant window (client-side) but GetItem() would return null (server-side).

Research angle: How does EQEmu's content filter decide which items to exclude? Check the items table for any expansion-related columns. The column min_expansion didn't exist in our schema — what column DOES v23.10.3 use? Could it be in the item ID range itself, or in a separate content_flags table?

-- Check if there's a content filtering table SHOW TABLES LIKE '%content%'; -- Check all columns that might relate to expansion filtering SELECT COLUMN_NAME FROM INFORMATION_SCHEMA.COLUMNS WHERE TABLE_SCHEMA='peq' AND TABLE_NAME='items' AND COLUMN_NAME LIKE '%exp%' OR COLUMN_NAME LIKE '%content%' OR COLUMN_NAME LIKE '%era%' OR COLUMN_NAME LIKE '%flag%';

Question 4
Does database.GetItem() fail for certain items in shared memory?

The packet logs confirm the exact failure: OP_ShopPlayerBuy → OP_SimpleMessage (already sold) → OP_ShopDelItem → OP_ShopEndConfirm. This matches the code path where database.GetItem(item_id) returns null.

The item EXISTS in the MySQL items table. Shared memory was rebuilt fresh. Yet GetItem() can't find it. This means either: (a) shared_memory.exe skipped loading the item, or (b) the lookup key/index doesn't match.

Research angle: Look at EQEmu's shared_memory.cpp — does it load ALL items from the DB, or does it filter? Check common/shared_tasks.cpp or common/items.cpp for the loading query. Does it have a WHERE clause that excludes certain items?

Question 5 — Strong Lead 🔥
UNK132 field length mismatch — 19 vs 20 characters

Comparing working vs broken items in the database:

Working items (Short Sword 5001, Water Flask 13006): UNK132 = '00000000000000000000' (20 characters)
Broken items (Soul Devourer 11001, Big Soul Devourer 11000, Ancient Frostreaver 32801): UNK132 = '0000000000000000000' (19 characters)
Bags (Annihilator's Satchel 139831): UNK132 = NULL

If shared_memory.exe parses UNK132 as a fixed-width binary field and expects exactly 20 characters, a 19-character value could cause a parsing error that makes the item fail to load — or corrupt the index so GetItem() can't find it.

-- Check UNK132 lengths across ALL our merchant items SELECT i.id, i.Name, LENGTH(i.UNK132) as unk132_len, i.UNK132 FROM items i JOIN merchantlist ml ON ml.item = i.id WHERE ml.merchantid IN (104901,104902,104903,104908,104909,104910) ORDER BY LENGTH(i.UNK132), i.id; -- Quick fix attempt: pad all to 20 chars UPDATE items SET UNK132 = '00000000000000000000' WHERE id IN ( SELECT item FROM merchantlist WHERE merchantid IN (104901,104902,104903,104908,104909,104910) ) AND (LENGTH(UNK132) != 20 OR UNK132 IS NULL);

Question 6
Why didn't Trading/Merchant log categories produce output?

We enabled verbose logging: set log_to_file=3 and log_to_gmsay=3 for Trading (category 29) and Merchants (category 67) in logsys_categories. Ran #logs reload in-game. No merchant-level debug messages appeared in logs.

Only packet-level logs were visible in the zone log file. The Trading/Merchant categories may not have any log statements in v23.10.3, or the category IDs may have shifted between versions.

Research angle: Check if EQEmu v23.10.3 actually has LogTrading or LogMerchants calls in the merchant buy handler. If not, those categories are useless and we need packet-level debugging or custom source code builds.

Question 7
Could the item ID range affect shared memory loading?

Working items: Short Sword (5001), Water Flask (13006) — low IDs from base EverQuest.
Broken items: Soul Devourer (11001), Ancient Frostreaver (32801), items up to 147494 — spanning many expansion ranges.

Shared memory uses a binary file (C:\Users\deacyde-minipc\server\shared\items, 157MB). If it uses a fixed-size array indexed by item ID, there could be an upper bound. Or if it uses a hash table, collisions could cause certain IDs to be unreachable.

Counter-evidence: Bags work and include high IDs like 139831. But bags might use a different lookup path.
